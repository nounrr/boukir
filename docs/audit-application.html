Audit de l’application Bpukir

Cartographie du fonctionnement, contrôle des requêtes et routes, points faibles confirmés dans le code, puis solutions classées par urgence.

Audit du 10 juillet 2026 React 19 + TypeScript + Vite Express 4 + MySQL Analyse statique et build local

Verdict : corrections critiques requises

L’application possède une couverture fonctionnelle riche et le build de production réussit. Elle ne doit toutefois pas être exposée publiquement dans son état actuel : des API IA sont publiques et modifient la base, tandis que plusieurs opérations d’administration ne vérifient aucun rôle côté serveur.

Mise à jour du 10 juillet 2026 : les deux points P0 ont été corrigés dans le code : les routes IA exigent désormais un rôle autorisé et les API d’administration sensibles appliquent un contrôle de rôle côté serveur. Un redémarrage de l’API est requis pour activer ces changements.

Mise à jour P1 : secret JWT obligatoire, limites de connexion et uploads, fichiers employés privés, routes statiques accessibles, socket réparé et dépendances mises à jour. L’audit de production ne conserve qu’une alerte haute sur xlsx, sans correctif publié par npm ; ses imports sont limités à 10 Mo et réservés aux rôles de gestion.

Mise à jour P2 : le contexte d’audit ne fait plus confiance à x-user-id, les origines CORS sont configurables par liste blanche, Helmet et des limites de requête sont actifs, et la règle de mot de passe reste appliquée jusqu’au changement effectif durant la semaine.

Fonctionnement logique de l’application

1. Interface ReactLes 59 pages affichent stock, contacts, bons, paiements, caisse, rapports, e-commerce et administration.
2. État et sessionRedux Toolkit / RTK Query gèrent les appels. Redux Persist et localStorage conservent le JWT et plusieurs données métier.
3. Requête HTTPLe frontend appelle /api/*. En développement, Vite proxifie vers l’API sur le port 3001.
4. API ExpressCORS, contexte d’audit, authentification JWT globale, politique de mot de passe, puis routeur métier.
5. MySQL / servicesLes routeurs lisent ou modifient MySQL. Socket.IO pousse les notifications ; OpenAI et WhatsApp sont appelés selon le module.
Exceptions publiques prévues : login, catalogue e-commerce, recherche, promo, points de retrait, catégories/marques en lecture, hero slides, checkout invité et fichiers /uploads. Exception non sûre : tout /api/ai est aussi public.

Cycle d’une requête authentifiée

  1. Le JWT est lu depuis Redux/localStorage et envoyé dans Authorization: Bearer ….
  2. Le middleware global vérifie la signature et place l’identité dans req.user.
  3. La politique de changement de mot de passe du lundi peut bloquer la requête.
  4. Le routeur valide partiellement les paramètres, exécute la requête SQL et renvoie du JSON.
  5. RTK Query met à jour son cache ; Socket.IO peut notifier les utilisateurs concernés.

Points solides observés : requêtes SQL généralement paramétrées, bcrypt pour les mots de passe, transactions présentes dans plusieurs workflows financiers, vérification du JWT côté socket, gestionnaire d’erreur central et build de production fonctionnel.

Points faibles confirmés et solutions

P0 critiqueP1 majeurP2 importantP3 amélioration
P0

Les API IA coûteuses et modificatrices sont publiques

Le middleware autorise sans JWT tout chemin commençant par /api/ai. Or les routes de traduction et de génération appellent OpenAI, et plusieurs utilisent commit = true par défaut avant d’exécuter des UPDATE sur produits et catégories. Un visiteur anonyme peut donc consommer le budget API et altérer le catalogue.

Preuves : backend/index.js:219-220 ; backend/routes/ai.js:734, 784, 795, 1250-1251, 1523, 1532, 1605-1606, 1871, 2080.
Solution : retirer l’exception publique ; appliquer verifyToken + requireRoles('PDG','Manager') au routeur entier ; limiter les modèles autorisés, la taille des lots et les appels par utilisateur/IP ; mettre commit=false par défaut et journaliser le coût.
P0

Contrôle des rôles absent sur des API d’administration

Le frontend cache des pages via ProtectedRoute, mais cela n’est pas une barrière de sécurité. Côté API, tout employé possédant un JWT peut créer un autre employé PDG, changer rôles et mots de passe, consulter les salaires, modifier les horaires d’accès ou lire les journaux d’audit.

Preuves : backend/routes/employees.js:17-148 sans garde de rôle ; backend/routes/employe_salaires.js:74-364 ; backend/routes/accessSchedules.js:8-448 vérifie seulement le token ; backend/routes/audit.js:59-242 sans garde PDG.
Solution : définir une matrice RBAC serveur par ressource/action, puis protéger les routeurs au niveau router.use(requireRoles(...)). Ignorer created_by/updated_by reçus du client et toujours utiliser req.user.id. Ajouter des tests d’autorisation 401/403 pour chaque rôle.
P1

Secret JWT de secours connu et durée de session longue

Si JWT_SECRET manque, le serveur signe et accepte les tokens avec dev-secret. Une mauvaise configuration de production suffit alors pour fabriquer un token PDG. La durée par défaut est de 7 jours et aucun mécanisme de révocation n’est prévu.

Preuves : backend/middleware/auth.js:10, 24-26 ; backend/routes/users.js:37-38 ; backend/socket/socketServer.js:43.
Solution : refuser le démarrage si le secret est absent ou faible ; séparer les audiences employé/e-commerce ; réduire la durée du token d’accès, ajouter refresh token rotatif et invalidation à la suppression/changement de rôle.
P1

Documents employés et justificatifs exposés publiquement

Tout /uploads/* contourne l’authentification et le dossier complet est servi statiquement. Cela inclut /uploads/employee_docs et les images de paiements. Une URL obtenue par historique, journal, cache ou partage donne accès au fichier sans connexion.

Preuves : backend/index.js:217-218, 256-258 ; backend/routes/upload.js:10-12, 55-59.
Solution : ne publier que les images catalogue. Placer les pièces privées hors du répertoire statique et fournir un endpoint de téléchargement protégé par rôle/propriétaire, avec nom opaque et en-têtes Content-Disposition, nosniff et cache privé.
P1

Vulnérabilités connues dans les dépendances de production

npm audit --omit=dev trouve 29 vulnérabilités : 1 critique, 16 hautes, 11 modérées et 1 faible. Les dépendances directes concernées incluent notamment jsPDF, Axios, Express, Multer, React Router DOM et xlsx.

Résultat local du 10/07/2026 : 29 alertes sur 478 dépendances de production. Multer est en 2.0.2 ; React Router DOM en 7.7.1 ; xlsx en 0.18.5.
Solution : créer une branche dédiée, mettre à jour par petits lots avec tests de régression, commencer par Multer/Express/Axios/jsPDF/React Router. Remplacer ou isoler xlsx si aucune version npm corrigée n’est disponible, puis automatiser audit et Dependabot/Renovate.
P1

Uploads insuffisamment bornés et validés

Les uploads génériques de produits n’ont ni limite de taille ni filtre de type. Les documents employés acceptent le MIME déclaré par le client et sont explicitement de taille illimitée. Cela permet épuisement disque/mémoire et fichiers déguisés.

Preuves : backend/routes/products.js:13-32 ; backend/routes/upload.js:38-46. La dépendance Multer installée possède aussi des alertes DoS.
Solution : limites globales et par route (taille, nombre de fichiers, champs), détection du type réel par signature, réencodage des images, quotas, nettoyage en cas d’échec, stockage objet et antivirus pour PDF/documents.
P1

Absence de limitation sur le login employé

Le login e-commerce gère les tentatives et le verrouillage, mais /api/auth/login pour les employés ne possède ni limitation par IP/CIN ni verrouillage progressif. Les identifiants peuvent être testés en continu.

Preuves : backend/routes/auth.js:27-77 ; le verrouillage existe seulement dans backend/routes/users.js:340-395.
Solution : rate limit distribué (Redis si plusieurs instances), délai progressif, compteur par IP + identifiant, alerte après seuil, messages uniformes et MFA pour PDG/Manager.
P1

Routes Express inaccessibles à cause de l’ordre

GET /employees/deleted/list est déclaré après GET /employees/:id : “deleted” est traité comme un ID. Le même défaut touche GET /products/last-commandes, placé après GET /products/:id.

Preuves : backend/routes/employees.js:24 puis 123 ; backend/routes/products.js:1857 puis 3211.
Solution : déclarer toutes les routes statiques avant /:id et contraindre l’ID avec une validation numérique. Ajouter un test de disponibilité de toutes les routes déclarées.
P1

Handler Socket.IO “ping” défectueux

Le callback commence par nom_complet || socket.user.socket.emit(...). nom_complet n’est pas défini ; recevoir l’événement peut déclencher une erreur d’exécution au lieu de répondre “pong”.

Preuve : backend/socket/socketServer.js:120-122. Le fichier passe le parseur Node, mais le chemin est incorrect à l’exécution.
Solution : remplacer le corps par socket.emit('pong', { timestamp: Date.now() }) et ajouter un test socket authentifié connexion/ping/déconnexion.
P2

Traçabilité falsifiable sur les routes publiques

Le contexte d’audit accepte directement l’en-tête client x-user-id. Les routes protégées le remplacent après vérification du token, mais les écritures publiques (par exemple checkout invité) peuvent attribuer une opération à un ID employé choisi par l’appelant.

Preuves : backend/index.js:133-138, 242-247 ; backend/db/pool.js:30-33.
Solution : ne jamais faire confiance à x-user-id venant d’Internet ; l’identité doit provenir exclusivement du JWT validé. Pour un invité, stocker null et un identifiant guest séparé.
P2

Politique de mot de passe incohérente et “fail open”

Le changement est imposé uniquement le lundi. Un utilisateur qui ne se connecte pas ce jour-là n’est plus bloqué mardi. En cas d’erreur base, le middleware autorise l’accès. Changer chaque semaine incite aussi aux mots de passe prévisibles.

Preuves : backend/middleware/passwordPolicy.js:23-39, 67-81.
Solution : préférer longueur forte, mots de passe compromis, MFA et rotation seulement en cas de risque. Si une règle périodique reste obligatoire, enregistrer une échéance persistante et la faire respecter tous les jours ; échouer de façon contrôlée sur les opérations sensibles.
P2

CORS trop permissif et protections HTTP absentes

L’API reflète toute origine avec credentials: true. Aucun Helmet, HSTS, CSP ou limite globale de taille JSON n’est configuré. Le JWT dans l’en-tête réduit le risque CSRF classique, mais cette configuration élargit inutilement la surface.

Preuves : backend/index.js:122-131 ; aucune dépendance Helmet/rate-limit globale dans package.json.
Solution : liste blanche d’origines par environnement, Helmet, limite JSON/URL-encoded, trust proxy explicite, TLS obligatoire et politique CSP adaptée au frontend.
P2

Qualité non bloquante : 3 945 alertes ESLint

Le lint échoue avec 3 818 erreurs et 127 avertissements. Au-delà des nombreux any, un Hook React est appelé conditionnellement dans AvoirFormModal, ce qui peut associer le mauvais état entre deux rendus. Une règle ESLint Sonar référencée n’est même pas installée.

Commande : npm run lint. Exemple fonctionnel : frontend/src/components/AvoirFormModal.tsx:138. Configuration cassée : backend/routes/employees.js:63.
Solution : corriger d’abord les règles fonctionnelles (Hooks, dépendances d’effets, variables non définies), établir une baseline pour les any, puis réduire progressivement. Bloquer seulement les nouvelles erreurs en CI durant la transition.
P2

Bundle frontend lourd et état dupliqué

Le bundle principal généré fait 3,88 Mo minifié (1,03 Mo gzip). L’application importe beaucoup de pages sans découpage. Le store Redux est en plus fourni deux fois : dans main.tsx et dans App.tsx. Plusieurs grandes collections métier sont persistées localement, avec risque de données obsolètes et d’exposition en cas de XSS.

Build Vite local ; frontend/src/main.tsx:22-23 ; frontend/src/App.tsx:755-759 ; frontend/src/store/index.ts:28-31.
Solution : un seul Provider ; lazy loading des pages et bibliothèques PDF/XLSX ; ne persister que la session minimale ; préférer cookie HttpOnly sécurisé pour l’authentification et cache RTK Query borné/invalidation serveur.
P3

Migrations de schéma lancées de façon asynchrone au démarrage

Les fonctions ensure…Schema démarrent avant l’écoute, mais ne sont pas chaînées à ensureDb(). Le serveur peut donc accepter des requêtes alors qu’une migration continue ou a échoué, et plusieurs instances peuvent tenter le même DDL.

Preuve : backend/index.js:88-116 puis démarrage indépendant à 424-430.
Solution : migrations versionnées exécutées comme étape de déploiement avec verrou ; démarrage refusé si la version attendue n’est pas présente ; retirer les ALTER TABLE automatiques des routes et du boot.

Tests à ajouter en priorité

  • RBAC : matrice invité / Employé / Chauffeur / Manager / PDG × lecture / création / modification / suppression.
  • Documents : un utilisateur non autorisé ne peut ni lister ni télécharger une pièce employé ou paiement.
  • Stock et finance : double soumission, annulation répétée, paiement partiel, avoir, remise, stock insuffisant et concurrence.
  • Routes : test “smoke” de chaque endpoint statique afin de détecter l’interception par /:id.
  • IA : authentification, rôle, quota, lot maximal, modèle autorisé, preview sans écriture et commit explicite.
  • Uploads : extension mensongère, MIME incorrect, fichier trop grand, requête interrompue, trop de fichiers et nettoyage.
  • Frontend : navigation par rôle, expiration du token, changement de mot de passe, erreurs API, rendu mobile et chargement paresseux.

Conclusion

La logique générale est compréhensible et plusieurs workflows utilisent de bonnes bases techniques. Le risque principal n’est pas le SQL, mais la frontière d’autorisation : l’interface applique des rôles que l’API ne réplique pas systématiquement. La priorité absolue est donc de sécuriser le serveur, puis de protéger les fichiers et les services coûteux avant toute optimisation visuelle ou refactorisation.

Confiance de l’audit : élevée pour les constats accompagnés de lignes de code ; moyenne pour l’impact métier exact, qui doit être confirmé avec une base de test représentative et les règles métier officielles.
